Companion concept  ·  can stand alone or fold into SonOfAnton

Trackbed_

A deliberately thin roadmap + status layer that gives Superpowers the one thing it lacks — a route — without the weight of GSD. Keep the rails. Lose the train.

Origin: the GSD/Superpowers split lived on Panviva Core Status: idea — pre-build Relation: may be SonOfAnton's orchestration backbone

01 The metaphor

🚆 GSD — the train / tank

Heavy. Slow to start and stop. Lots of ceremony to get rolling. But it runs on rails and carries structure — you always know where it is on the line and what's coming. Great at not losing the route across 17 phases; terrible at darting around.

vs

🏎 Superpowers — the racing car

Fast, responsive, you feel the road, change line instantly — TDD, brainstorming, debugging, all light and reactive. But it has no idea where the track goes. Drop it with no map and it'll run a brilliant lap of the wrong circuit.

★ The move: rails without the train

Don't pick one. Keep the route + manifest (roadmap + status), throw away the locomotive (GSD's agents, discuss/plan/execute ceremony, command sprawl), and put the racing car on the rails for each leg. The rails say which segment is next and what "done" means; the car drives that segment flat-out.

02 Why this exists

Tried GSD — too heavy (train/tank). Needed the resilience of Superpowers (racing car). But the one thing worth keeping from GSD is its roadmap + status: organizing stories within an epic, ordering, dependencies, "what's owed."

This pattern is already proven by hand on Panviva Core — the project's own rule: "GSD = state and roadmap management ONLY. Superpowers = phase-level execution. These two do not overlap." It ran 11+ phases that way. Trackbed just makes that half thin and owned instead of carried on GSD's back.

Prior art check No polished off-the-shelf tool does exactly this (lightweight roadmap/status + Superpowers-per-phase). The two ecosystems come from opposite philosophies — GSD is all-in on planning machinery; Superpowers is deliberately unopinionated about project structure. The clean bridge between them isn't shipped. So: tailor it — but it's small.

03 What's actually valuable (it's tiny)

Strip GSD to what gets used and you're left with two files and three conventions:

ROADMAP — the rails

Ordered phases. Each = a Jira key + one-line scope + depends-on. That's the route the car follows.

STATE — the signals

Current phase, status, what's owed (gates not yet run). The "you are here + what's red" board.

Three conventions: phase numbering (incl. decimals like 11.2 for insertions) · dependency edges · an explicit "done = X" per phase. Everything else in GSD is the heaviness being rejected.

epic PANV-60446 ▸ 11.1 ✓ 11.3 ✓ 11.2 ◉ current 11.4 → next 13 ⛔ owes E2E 12 …17
// the rails, drawn: done · current · next-unblocked · blocked-with-reason · not-yet. This is all Trackbed needs to know to hand the racing car its next segment.

04 Options — cheapest → richest

1 · Cherry-pick GSD

Keep only the state/roadmap skills + .planning/ files; never invoke plan/execute. ~Already doing this. Cost zero — but still carries GSD's file format + quirks.

2 · Thin bespoke layer lean pick

Drop GSD. Own a simpler two-file format (no gsd_state_version cruft) + a tiny helper: read "next unblocked phase" → hand to Superpowers brainstorming/executing-plans. A few hundred lines, fully yours, debuggable, no heaviness.

3 · Backbone of SonOfAnton

Same data model the app already needs for its status strip + button-gating. "Next phase / whose turn / what's owed" is what a control panel reads. Build once, use in both.

05 The thin format (sketch)

Illustrative only — the point is how little it is. One roadmap, one state pointer.

# roadmap.yml — the rails
epic: PANV-60446
phases:
  - id: 11.2   jira: PANV-61955   done: code+gates   depends: [11.1]
    scope: "BFF API-key validation — remove Identity HTTP fallback"
  - id: 11.4   jira: PANV-62093   status: next        depends: [11.3]
    scope: "RedisContainerMapWriter — restore Marten writer (OCP)"
  - id: 13     jira: PANV-61405   status: blocked     depends: [11.2]
    owes: [e2e, shell-regression]        # a red signal

# state.yml — the signals (one pointer + what's owed)
current: 11.2
turn: architect            # whose move (SonOfAnton reads this)
owed: { 11.2: [e2e, shell-regression] }

A ~few-hundred-line helper answers exactly three questions: what's the next unblocked phase? · what does this phase owe before it's done? · hand this phase to Superpowers and mark it in-progress. That's the whole engine.

05.5 Visualization — QML brings the roadmap to life the fun part

The roadmap is just data (§05). QML is a graphics/animation engine first, UI toolkit second — so instead of GSD's flat markdown, Trackbed draws the roadmap as interactive graphics. Same data, multiple lenses.

Core principle — two views, one model The DAG view and the Gantt view are the same roadmap rendered two ways. You're not adding data, you're adding lenses. And both are the same component SonOfAnton's status strip uses — Trackbed's roadmap view is the app's "where are we" board, scaled up. Build the model once; the visuals are skins on it.

Lens A — the DAG / rail view v0 — data you already have

Phases as nodes, dependencies as edges, status as color (palette already maps it). Answers the day-to-day question: what blocks what · what's the critical path · what's parallelizable.

  • Click a node → open the Jira ticket — Qt.openUrlExternally("…/browse/PANV-61955"). The roadmap becomes a launchpad, not a picture. (Just needs the key + base URL — no API.)
  • Hover → tooltip with scope, what's owed, dependencies — the STATE data surfaced visually instead of buried in a file.
  • Edges are the real win: markdown says "11.4 depends-on 11.3"; a drawn DAG shows it — blockers, parallel legs, critical path, at a glance. (We leaned on the phase table constantly this epic — this replaces that.)
  • Live, not a snapshot: nodes pulse/animate as state changes (current = amber glow, a phase clearing its gates flips green). QML animations are essentially free.
Design call — layout A roadmap-with-dependencies is a DAG, and auto-laying-out a DAG nicely is non-trivial. Start with manual coords (x/y per phase in the roadmap file) — at ~17 nodes, hand-placement looks better than any auto-layout and costs a fraction of the work. Topological/layered auto-layout is a "later, if it earns it" feature, not v0.

Lens B — the Gantt view v1 — needs a time source

Same phases on a time axis — positioned bars sized by duration. Answers a different question: when · how long · are we on track. Toggle rail ⇄ Gantt on the same roadmap. (QML draws this trivially — it's positioned/sized Rectangles on an axis.)

The honest catch — where does TIME come from? The DAG needs only order + dependencies (you have these). A Gantt needs a time dimension you don't have yet — start/end/duration per phase. Three sources, increasing cost:
SourceWhat it givesCost
Story points → durationA projected timeline (pts × velocity, laid out by dependency order). Clearly labelled "estimated."Cheap — you already estimate. Zero new upkeep.
Manual datesExplicit start/end in the roadmap file.Simple, you control it — but hand-maintained.
Pull from JiraSprint dates + created/resolved = an actuals Gantt.Big lift — Atlassian API, auth, rate limits. Later phase.
A Gantt with fake-confident dates is worse than no Gantt. Each step must be honest about its precision — "projected" vs "actual."

Sequencing — keep each step honest

StepViewFed byClaim it can honestly make
v0DAG / railOrder + dependencies have it"Here's the structure — what's next, what's blocked." + Jira deep-links.
v1+ GanttStory points → duration"Here's the projected timeline" (estimated, not actuals).
v2Both, Jira-syncedAtlassian API later"Here's reality" — live ticket status + real dates. One "connect to Jira" feature (status + dates share the auth work).

v0 shows structure from data you have; v1 projects time from estimates you have; v2 reflects reality from Jira when it's worth the API cost. No step pretends to precision it lacks. This is the "platform we add tools onto over time" — a view layer that grows.

06 The one caution the metaphor surfaces

The racing car runs the wrong track happily Superpowers' weakness is it'll execute a great lap of the wrong circuit — drift, carried stale baselines, lost route between segments (exactly what bit us when stale state crept in). So Trackbed's most important job isn't speed — it's being thin but authoritative: the single source of "where we are / what's next / what's done," that the car can't override. Rails the car can't jump.

07 Relation to SonOfAnton

Trackbed can ship standalone (CLI + two files, drives Superpowers in a plain terminal) or become SonOfAnton's orchestration backbone — the app's status strip and button-gating are literally Trackbed's signals rendered in QML. Build it standalone first (option 2), and if the app happens, it reads the same files. Nothing wasted either way.
